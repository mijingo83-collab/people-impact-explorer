# People Impact Explorer 고정 작업 지침

## 도구의 목적

이 프로젝트는 국내외 주요 이슈가 인사 업무에 미치는 영향을 빠르게 판단하기 위한 `인사 시사점 분석 레포트`이다. 기준 기사와 추가 리서치 자료를 함께 검토해 People 기능별 영향도, 상위 리스크, 근거 자료, 권고 액션을 하나의 HTML 리포트로 제공한다.

리포트의 목적은 단순 뉴스 요약이 아니라 인사 조직이 바로 확인해야 할 질문, 리스크, 내부 지표, 실행 과제를 정리하는 것이다.

## People 기능 9개와 세부 정의

1. 조직/임원

   * 조직개편
   * 임원 지원 및 관리
   * 임원 인사
   * 차세대 리더 양성 및 관리

2. 인력운영

   * 인력 운영 계획 수립 및 실적 관리
   * 인력 충원 및 효율화
   * 휴직/복직/퇴직 관리

3. 제도

   * 승격
   * 평가
   * 연봉 산정
   * 시상/징계

4. 채용

   * 외부 인력 발굴 및 채용
   * 우수 인력 선확보
   * 산학교류 및 참여연구원 관리

5. 교육

   * 직급/직무별 특화 교육
   * 차세대 리더 육성
   * AI/직무/리더십/어학 교육

6. 노사

   * 대내외 동향 관리
   * 특이 인력 파악

7. 조직문화

   * 캠퍼스 행사 운영
   * 일하는 문화 조성

8. 총무

   * 의전
   * 레이아웃 관리
   * 사무환경/비품
   * 식당
   * 인허가
   * 차량 관리

9. 보안

   * 보안 시스템 개선 및 관리
   * 보안 점검

## 고정 섹션 순서

리포트 섹션 순서는 `outputs/people-impact-explorer.html` 및 `outputs/template.html`의 구조를 따른다. 임의로 추가, 삭제, 재배열하지 않는다.

1. 상단 헤더: 레포트명, 상태 보드
2. 분석 대상 기사와 인사 판단 요약
3. 상위 리스크 3개
4. 영향 영역 순위와 상세 분석
5. 근거 자료와 권고 액션
6. 유사·참고 사례
7. 하단 보고서 안내 문구

상세 분석 내부 순서도 고정한다.

1. 관련 업무 범위
2. 직접 영향
3. 영향 전개 경로
4. 2차 영향
5. 검토 필요 리스크
6. 왜 영향이 있는가
7. 핵심 시사점
8. 확인해야 할 내부 지표

상단 영향 전개 경로 요약 카드의 순서도 고정한다.

1. 기준 기사
2. 직접 영향
3. 2차 영향
4. 검토 필요 리스크
5. 우선 검토

## 디자인 유지 원칙

* `outputs/people-impact-explorer.html`의 현재 상태를 기준 디자인으로 본다.
* `outputs/template.html`은 원본 템플릿이며 자동화 작업에서 수정하지 않는다.
* 새 리포트는 `outputs/template.html`을 복사하거나 그 구조를 기반으로 별도 파일에 생성한다.
* 화면에 표시되는 상단 H1/레포트명은 `인사 시사점 분석 레포트`를 사용한다.
* `인사 시사점 분석 도구`라는 명칭으로 되돌리지 않는다.
* 섹션 순서, 카드 스타일, 글자 크기, 여백, 색상 체계, 영향 수준 배지 체계를 임의로 변경하지 않는다.
* CSS 클래스명, 레이아웃 그리드, 카드 구성, 배지 표현 방식, 버튼/링크 스타일을 디자인 개선 명목으로 바꾸지 않는다.
* 이후 작업의 범위는 콘텐츠 갱신, 데이터 주입, 데일리 리포트 생성 구조 작성으로 제한한다.
* 단, 6개 헤더 탭 표시를 위해 기존 탭 영역이 깨지는 경우에는 기존 디자인 체계를 유지하는 범위에서 줄바꿈 또는 가로 스크롤 등 최소한의 반응형 처리만 허용한다.

## 최신 템플릿 구성 기준

현재 최신 템플릿은 `outputs/template.html`, `index.html`, `archive.html`, `outputs/reports.json`이 함께 동작하는 구조다.

* `outputs/template.html`은 개별 데일리 리포트 화면의 기준 템플릿이다.
* `outputs/latest.html`은 최신 리포트 고정 URL용 파일이며, 매일 생성되는 날짜별 HTML과 같은 내용으로 갱신한다.
* `outputs/데일리_인사시사점리포트_YYYYMMDD.html`은 날짜별 보관용 HTML이다.
* `index.html`은 공개 사이트의 첫 화면이며, 최신 리포트와 이력 달력으로 이동하는 진입점이다.
* `archive.html`은 날짜별 이력 달력 화면이며, `outputs/reports.json`을 읽어 날짜별 리포트 목록을 표시한다.
* `outputs/reports.json`은 달력 화면의 데이터 원본이다. 새 리포트 생성 시 반드시 같은 날짜 항목을 추가하거나 갱신한다.
* 개별 리포트 상단 상태 보드는 `자료 유형`, `분석 관점`, `대상 이슈`, `이력 보기 / 달력 열기`의 4칸 구성을 유지하며, 달력 링크 대상은 `../archive.html`이다.
* 달력 화면에서 사용자가 날짜를 선택하면 해당 날짜의 6개 헤더별 이슈와 날짜별 HTML 링크를 확인할 수 있어야 한다.
* 새 템플릿 변경은 위 파일들의 연결 관계를 깨뜨리지 않는 범위에서만 수행한다.
* 템플릿 구조를 변경할 경우 `outputs/template.html`, `outputs/latest.html`, 당일 날짜별 HTML, `archive.html`, `outputs/reports.json`의 동작을 함께 확인한다.

## 기사 헤더 카테고리 기준

데일리 리포트의 기준 기사는 `업계`, `산업`, `관계사`, `정책`, `경제`, `글로벌` 6개 헤더별로 각 1건씩 선정한다.

각 헤더는 사용자가 관심 영역을 선택해 확인할 수 있는 탐색 단위이며, 모든 기준 기사는 People 기능 9개와 연결해 인사 시사점 분석을 제공한다.

다만 6개 헤더는 동일한 중요도를 의미하지 않는다. 영향도 정렬, 상위 리스크, 권고 액션의 우선순위는 헤더 순서가 아니라 삼성디스플레이 People 업무에 미치는 영향 수준을 기준으로 판단한다.

사용 가능한 헤더 키워드는 아래 6개로 고정한다.

1. 업계

   * 디스플레이 업계 이슈에만 사용한다.
   * OLED, LCD, 마이크로LED, QD, IT·모바일·TV·전장 디스플레이, 패널 수급, 투자, 감산, 증설, 고객사 수요, 경쟁사 동향, 기술 전환, 핵심 인재 경쟁 등 디스플레이 산업과 직접 연결된 기사에 부여한다.
   * 삼성디스플레이 People 업무와의 직접성이 가장 높은 후보군으로 우선 검토한다.
   * 디스플레이와 직접 관련 없는 반도체, 배터리, 전자부품, 일반 제조업 이슈에는 업계 헤더를 사용하지 않는다.

2. 산업

   * 디스플레이를 제외한 인접 첨단산업 및 기술 산업 이슈에 사용한다.
   * 반도체, 배터리, AI 제조업, 첨단소재, 전자부품, 전장, 제조 자동화 등 삼성디스플레이 People 업무에 참고 가치가 있는 산업 이슈에 한정한다.
   * 산업 헤더는 넓은 산업 동향을 채우기 위한 슬롯으로 사용하지 않는다.
   * 단순 산업 동향, 실적, 주가, 일반 경영 뉴스는 선정하지 않는다.
   * 채용, 인력운영, 교육, 보상, 조직개편, 노사, 보안 등 People 업무 영향이 명확한 경우에만 선정한다.

3. 관계사

   * 삼성전자, 삼성SDI, 삼성전기, 삼성SDS 등 삼성 관계사의 공개 언론 기사에만 사용한다.
   * 조직개편, 임원 인사, 채용, 보상, 성과급, 복지, 리텐션, 노사, 교육, 보안, 사업 재편 등 People 업무에 영향을 줄 수 있는 이슈를 포함한다.
   * 타 대기업, 경쟁사, 협력사, 고객사, 일반 국내 기업 이슈에는 관계사 헤더를 사용하지 않는다.
   * 공개 언론 기사 기반 내용만 사용하며, 삼성디스플레이 내부 상황으로 직접 단정하지 않는다.

4. 정책

   * 국가정책, 산업정책, 노동정책, 고용정책, 교육정책, 지역정책, 세제·규제, 고용·노동 관련 법제도 변화에 사용한다.
   * 정부 발표 자체보다 People 업무에 영향을 줄 수 있는 신문·언론 기사 기반 이슈를 우선한다.
   * 정책 발표 기사라도 People 업무와 연결되는 내부 지표나 권고 액션을 도출하기 어려우면 선정하지 않는다.

5. 경제

   * 환율, 금리, 경기, 수출, 관세, 공급망, 원가, 실적, 투자심리, 고용시장 등 경영환경 변화에 사용한다.
   * 단순 경제 동향, 증시 해설, 거시 전망, 실적 전망 기사는 선정하지 않는다.
   * 채용, 인력운영, 보상재원, 조직개편, 노사 커뮤니케이션, 근무환경 비용 등 People 업무 영향이 명확한 경우에만 선정한다.

6. 글로벌

   * 해외 AI, 자동화, 인재경쟁, 직무 재편, 글로벌 기업의 채용·보상·교육·조직문화·보안 관련 이슈에 사용한다.
   * 국내 People 업무에 주는 시사점이 명확한 경우에만 선정한다.
   * 단순 해외 기업 실적, 제품 출시, 시장 점유율 기사는 선정하지 않는다.

사설/칼럼은 헤더 카테고리로 운영하지 않는다. 사설/칼럼은 기준 기사, 근거 자료, 유사·참고 사례로 사용하지 않고, 필요한 경우 이슈 감지 또는 쟁점 파악용 내부 참고로만 활용한다.

## 기사 헤더 표시 기준

데일리 리포트의 각 기준 기사 제목 앞에는 헤더 키워드를 대괄호 형식으로 표시한다.

사용 가능한 표시 형식은 아래 6개로 고정한다.

* `[업계]`
* `[산업]`
* `[관계사]`
* `[정책]`
* `[경제]`
* `[글로벌]`

기사 제목, 탭 제목, 분석 대상 기사명, `outputs/reports.json`의 `issues.title`에는 동일한 헤더 표시를 적용한다.

예시는 아래와 같다.

* `[업계] OLED 투자 경쟁 확대와 핵심 인력 수요 변화`
* `[관계사] 삼성전자 AI 인재 확보 관련 공개 이슈`
* `[정책] 첨단산업 인재양성 정책 확대`

헤더 키워드는 기사 성격을 빠르게 구분하기 위한 표시값이며, 기사 본문 제목 앞에만 사용한다. People 기능명, 리스크명, 권고 액션 제목에는 헤더 키워드를 반복 표시하지 않는다.

## 탭 표시 및 기본 선택 기준

데일리 리포트 화면에는 6개 헤더 탭을 아래 순서로 고정 표시한다.

1. 업계
2. 산업
3. 관계사
4. 정책
5. 경제
6. 글로벌

리포트 최초 진입 시 기본 선택 탭은 항상 `업계`로 설정한다.

사용자가 별도 탭을 클릭하지 않은 상태에서는 `[업계]` 기사와 해당 People Impact 분석이 첫 화면에 표시되어야 한다.

특정 실행일에 업계 헤더가 `당일 기사 기준 미충족`인 경우에도 업계 탭은 첫 번째 탭으로 유지한다. 이 경우 첫 화면에는 업계 기사 분석 대신 미충족 사유, 탐색한 출처 범위, 대체하지 않은 이유, 다음 확인 필요 사항을 표시한다.

탭 순서는 People 영향도 순위에 따라 변경하지 않는다. 영향도 정렬은 각 탭 내부의 People 기능 순위와 상위 리스크에서만 반영한다.

## 영향 수준 체계

영향 수준은 기능별 `score` 값을 기준으로 아래 체계를 유지한다.

* 92점 이상: `매우 높음`, CSS class `level-critical`
* 82점 이상 92점 미만: `높음`, CSS class `level-high`
* 68점 이상 82점 미만: `보통`, CSS class `level-medium`
* 68점 미만: `낮음`, CSS class `level-low`

영향도 정렬은 기사에 먼저 언급된 순서가 아니라 인사 조직이 먼저 검토해야 할 리스크가 큰 순서로 한다.

## 분석 원칙

* 기준 기사만 본 직접 영향과 추가 자료를 포함한 종합 판단을 구분한다.
* 각 이슈는 People 기능 9개 중 어디에 영향을 주는지 매핑한다.
* 각 헤더별 기사마다 People 기능 9개를 모두 표시한다.
* 영향 전개 경로는 기준 기사, 직접 영향, 2차 영향, 검토 필요 리스크, 우선 검토 항목으로 이어지도록 작성한다.
* `2차 영향`은 직접 영향 이후 인력운영, 제도, 교육, 노사, 보안 등 다른 People 기능으로 확산되는 후속 영향을 별도로 정리한다.
* 상위 리스크 3개는 현재 선택된 탭의 기준 기사에 대해 인사 조직이 먼저 확인해야 할 위험을 중심으로 선정한다.
* 리포트 전체 판단 요약에서는 6개 헤더 중 People 영향도가 높은 이슈를 우선적으로 언급할 수 있다.
* 근거 자료는 원문 기사, 신문·언론 기사, 보상 기사, 시장 기사 등 판단에 실제로 연결되는 기사 자료를 우선 사용한다.
* 산업통상부, 고용노동부, 정부 부처, 공공기관, 기업 공식 보도자료는 기준 기사·근거 자료·유사 사례의 주 출처로 사용하지 않는다. 필요한 경우에도 링크 버튼이 있는 근거 자료가 아니라 내부 확인 참고로만 다룬다.
* 추측성 표현은 줄이고, 확인 가능한 신호와 내부에서 검증할 지표를 함께 제시한다.
* 각 기능별 상세 분석에는 관련 업무 범위, 직접 영향, 영향 전개 경로, 2차 영향, 검토 필요 리스크, 영향 이유, 핵심 시사점, 확인해야 할 내부 지표가 들어가야 한다.
* 영향이 낮은 기능이라도 삭제하지 않는다. 낮은 점수, 낮은 영향 수준 배지, 짧은 설명을 통해 강약을 구분한다.

## 권고 액션 작성 원칙

* 권고 액션은 인사 조직이 바로 검토하거나 실행할 수 있는 문장으로 쓴다.
* 일반론이나 캠페인성 문구보다 점검, 정의, 비교, 모니터링, 설계, 기준 수립처럼 실행 동사가 분명한 문장을 사용한다.
* 선택된 People 기능의 업무 범위와 연결되도록 작성한다.
* 각 액션은 리스크나 내부 지표와 연결되어야 한다.
* 가능하면 기능별로 3개의 권고 액션을 제시한다.
* 영향도가 낮은 기능은 권고 액션을 간결하게 작성하되, 필수 필드는 비워두지 않는다.
* 권고 액션은 디자인 요소나 UI 수정 요청으로 확장하지 않는다.

## 링크 처리 원칙

* 기사 원문 보기, 자료 열기, 사례 열기는 각각 해당 항목의 실제 출처 URL을 사용한다.
* 기준 기사 URL을 근거 자료나 유사·참고 사례 링크에 반복 재사용하지 않는다.
* 각 이슈, 근거 자료, 유사·참고 사례는 독립적인 `url` 값을 가져야 한다.
* 실제 URL을 확인하지 못한 항목에는 링크 버튼을 만들지 않고 `링크 확인 필요`로 표시한다.
* 하나의 `defaultUrl` 또는 `articleUrl`을 모든 버튼에 일괄 재사용하지 않는다.
* 링크의 정확성은 리포트 신뢰도에 직접 영향을 주므로, 파일 생성 전 최소 1회 점검한다.

## 유사·참고 사례 작성 원칙

* 기존 섹션명 `유사 사례`는 `유사·참고 사례`로 사용한다.
* 유사·참고 사례는 현재 분석 내용을 반복하는 영역이 아니다.
* 현재 이슈의 파급 리스크와 권고 액션을 검증·보강하는 비교 근거 영역이다.
* 동일 기업, 동일 산업, 동일 이슈만 고집하지 않는다.
* 동일 산업의 유사 사례, 다른 첨단산업의 인재 확보/리텐션 사례, 국내 대기업의 인사/보상/채용/노사 사례, 정책·제도 변화 대응 사례, 해외 선행 사례를 필요에 따라 섞어 구성한다.
* 각 이슈 탭마다 최소 4개의 유사·참고 사례를 표시한다.
* 각 사례는 사례명, 사례 유형, 출처 또는 참고 범위, 왜 참고할 만한가, 인사 시사점, 연결 People 기능, 연결 권고 액션, 링크 여부를 포함한다.
* 실제 확인 가능한 사례가 부족할 경우 `비교 참고 사례`, `적용 가능성이 있는 선행 사례`, `추가 확인 필요 사례`로 구분한다.
* 근거 없는 사례를 실제 사례처럼 단정하지 않는다.
* 사례 카드는 compact하게 작성하고, 상위 1~2개 사례만 더 강조한다.
* 보상 사례는 보상 경쟁력 점검과 성과급 산식 설명자료 정비에, 노사 사례는 예상 문의·불만 쟁점 정리와 FAQ 작성에, 산학교류 사례는 장학생/참여연구원 전환율 점검에, 해외 AI 인재 사례는 핵심 직무 리텐션 인터뷰와 직무별 보상 밴드 점검에, AI 재교육 사례는 직무 전환 교육과 관리자 교육 과정 점검에, AI 보안 사례는 AI 사용 가이드와 보안 정책 업데이트에 연결한다.

## 뉴스 선정 기준

데일리 리포트의 기준 기사는 아래 6개 헤더별로 각 1건씩 선정한다.

1. 업계 이슈 1건

   * 디스플레이 업계 기사만 선정한다.
   * 삼성디스플레이 People 업무와의 직접성이 높은 이슈를 우선한다.

2. 산업 이슈 1건

   * 디스플레이 외 인접 첨단산업 및 기술 산업 기사 중 People 업무 영향이 명확한 기사만 선정한다.
   * 단순 산업 동향을 채우기 위한 용도로 선정하지 않는다.

3. 관계사 이슈 1건

   * 삼성 관계사의 공개 언론 기사만 선정한다.
   * 타 대기업, 경쟁사, 협력사, 고객사, 일반 국내 기업 이슈는 관계사 이슈로 선정하지 않는다.

4. 정책 이슈 1건

   * 국가정책, 산업정책, 노동정책, 고용정책, 교육정책, 규제·제도 변화 중 People 업무 영향이 있는 기사를 선정한다.

5. 경제 이슈 1건

   * 환율, 금리, 경기, 수출, 관세, 공급망, 원가, 고용시장 등 경제환경 변화 중 People 업무 영향이 명확한 기사를 선정한다.
   * 단순 시장 해설이나 전망성 기사는 선정하지 않는다.

6. 글로벌 이슈 1건

   * 해외 AI, 자동화, 인재경쟁, 직무 재편, 글로벌 HR 변화 중 국내 People 업무에 주는 시사점이 명확한 기사를 선정한다.

각 헤더별 기준 기사는 모두 실행일 당일 보도된 신문·언론 기사여야 한다. 실행일 당일 적합한 기사를 특정 헤더에서 확보하지 못한 경우, 과거 기사로 대체하지 않고 해당 헤더를 `당일 기사 기준 미충족`으로 표시한다.

기준 기사 6건 중 국내 이슈는 최소 5건이어야 하며, 글로벌 이슈는 최대 1건만 포함한다.

1차 탐색은 국내 주요 종합지를 우선한다: 조선일보, 중앙일보, 동아일보.

2차 보완은 경제지, 산업 전문지, 통신사 등 언론 기사만 활용한다: 한국경제, 매일경제, 연합뉴스, 전자신문, ZDNet Korea.

해외 매체는 글로벌 헤더 1건의 기준 기사 또는 국내 이슈의 보완 근거로만 사용한다.

해외 매체를 기준 기사 6건 전체에 반복 사용하지 않는다.

국내 1차 탐색 결과가 부족하더라도 정부 부처, 공공기관, 기업 공식 채널 자료를 기준 기사로 대체하지 않는다. 신문·언론 기사에서 People 영향도가 높은 다른 이슈를 재탐색한다.

최종 기사 선정은 헤더별 기사 확보 여부보다 People 영향도를 우선하되, 매일 리포트에는 6개 헤더의 상태가 모두 표시되어야 한다. 적합한 기준 기사를 찾지 못한 헤더는 임의 기사로 채우지 않고 `당일 기사 기준 미충족`으로 남긴다.

출처 기준 때문에 기사 품질이 떨어지면 안 되며, 최종 선정은 People 영향도가 가장 높은 이슈를 우선한다.

Samsung Display와 직접 관련된 내부 정보는 사용하지 않고, 공개 뉴스와 공개 자료만 사용한다.

리포트 생성 전 아래 출처 점검표를 통과해야 한다.

* 기준 기사 또는 미충족 상태가 6개 헤더 모두에 존재하는가
* 선정된 기준 기사의 보도일이 모두 실행일과 같은가
* 글로벌 기준 기사는 최대 1건인가
* 국내 기준 기사는 최소 5건인가
* 관계사 헤더는 삼성 관계사 공개 언론 기사만 사용했는가
* 업계 헤더는 디스플레이 업계 기사만 사용했는가
* 산업 헤더는 디스플레이 외 인접 첨단산업 중 People 영향이 명확한 기사만 사용했는가
* 경제 헤더는 단순 경제 동향이 아니라 People 업무 영향이 명확한 기사인가
* 조선일보, 중앙일보, 동아일보 1차 탐색을 먼저 수행했는가
* 1차 탐색이 부족한 경우 한국경제, 매일경제, 연합뉴스, 전자신문, ZDNet Korea 등 신문·언론 기사로 보완했는가
* 각 기준 기사와 근거 자료의 실제 URL이 확인되었는가
* 기준 기사 URL을 근거 자료나 유사·참고 사례 링크에 반복 재사용하지 않았는가

## 기존 리포트 중복 방지 기준

* 새 리포트의 6개 헤더별 기준 기사를 선정하기 전에 반드시 `outputs/reports.json`과 기존 `outputs/데일리_인사시사점리포트_*.html` 파일을 먼저 확인한다.
* 기존 리포트에서 이미 사용한 기준 기사 제목, 기준 기사 URL, 출처, 보도일, 이슈 키워드, 주요 기업명을 중복 점검 목록으로 만든다.
* 아래 중 하나에 해당하면 중복 이슈로 보고 기준 기사로 선정하지 않는다.

  * 기준 기사 URL이 기존 리포트와 같은 경우
  * 기사 제목이 같거나 사실상 같은 경우
  * 같은 기업, 같은 사건, 같은 보도일의 후속 없는 반복 기사인 경우
  * 기존 리포트의 분석 대상 이슈를 제목만 바꿔 다시 설명하는 경우
  * 기존 리포트의 근거 자료 또는 유사·참고 사례를 새 기준 기사처럼 승격해 재사용하는 경우
* 기존 이슈와 같은 기업 또는 산업을 다루더라도, 새로운 의사결정·공식 발표·정책 변화·노사 단계 변화·채용 공고 변화처럼 People 판단이 달라질 만한 새 사건이 있을 때만 후속 이슈로 선정할 수 있다.
* 후속 이슈를 선정할 경우 `sourceMix` 또는 판단 요약에 기존 이슈와 무엇이 달라졌는지 짧게 남긴다.
* 중복 여부가 애매하면 새 기준 기사로 쓰지 말고 근거 자료나 유사·참고 사례로만 사용한다.
* 국내 5건 이상, 글로벌 1건 이하 기준을 맞추더라도 기존 리포트와 중복되면 발행하지 말고 다른 후보를 재탐색한다.
* 특정 헤더에서 중복되지 않는 당일 적합 기사를 확보하지 못하면 과거 기사나 중복 기사로 대체하지 않고 `당일 기사 기준 미충족`으로 표시한다.
* 리포트 생성 전 아래 중복 점검표를 통과해야 한다.

  * `outputs/reports.json`의 기존 `issues` 목록과 새 6개 헤더별 이슈가 겹치지 않는가
  * 기존 날짜별 HTML의 기준 기사 URL과 새 기준 기사 URL이 겹치지 않는가
  * 기존 리포트의 근거 자료·유사·참고 사례를 새 기준 기사로 재사용하지 않았는가
  * 같은 사건의 단순 반복이 아니라 새 People 판단이 필요한 후속 이슈인가

## 데일리 리포트 생성 원칙

* 데일리 리포트는 반드시 `outputs/template.html`을 기반으로 생성한다.
* `outputs/template.html`은 디자인/구조 템플릿이며, 내부 샘플 콘텐츠는 고정 콘텐츠가 아니다.
* `outputs/template.html`에는 기사, 날짜, 출처, 분석 내용, 근거 자료, 권고 액션을 고정값으로 남기지 않고 placeholder로 둔다.
* 데일리 리포트 생성 과정에서는 `outputs/template.html` 자체를 수정하지 않고, placeholder를 실행일 기준 새 분석 결과로 치환한 별도 HTML 파일을 만든다.
* 매일 생성되는 결과물은 별도 HTML 파일로 저장한다.
* 자동화는 템플릿의 구조와 CSS를 재사용하고, 기사/분석/근거/액션 등 콘텐츠 데이터만 갱신한다.
* 데일리 리포트는 실행일 기준 업계, 산업, 관계사, 정책, 경제, 글로벌 6개 헤더별 기준 기사 또는 미충족 상태를 생성해야 한다.
* 매일 생성되는 리포트는 실행일 기준 뉴스 6건을 새로 선정하는 것을 원칙으로 한다.
* 특정 헤더에서 실행일 당일 보도된 적합한 신문·언론 기사를 확보하지 못한 경우, 과거 기사로 대체하지 않고 해당 헤더를 `당일 기사 기준 미충족`으로 표시한다.
* 데일리 리포트 화면에는 6개 헤더가 모두 접근 가능하도록 6개 탭을 표시한다.
* 각 탭은 해당 헤더의 기준 기사 1건에 대한 분석을 보여주되, 사용자가 다른 5개 헤더로 이동할 수 있는 탭 UI가 누락되면 안 된다.
* 리포트 최초 진입 시 기본 선택 탭은 항상 `업계`로 설정한다.
* 각 뉴스별 영향 영역 순위에는 People 기능 9개를 모두 표시한다.
* 영향이 큰 기능은 상위에 또렷하게 표시하고, 영향이 약한 기능도 삭제하지 말고 낮은 점수, 낮은 영향 수준 배지, 옅은 색상, 작은 보조 설명 등으로 강약을 구분한다.
* 영향이 약한 기능도 관련 업무 범위, 직접 영향, 영향 전개 경로, 2차 영향, 검토 필요 리스크, 영향 이유, 핵심 시사점, 확인해야 할 내부 지표를 확인할 수 있어야 한다.
* 특정 헤더가 `당일 기사 기준 미충족`인 경우에도 탭은 유지한다. 해당 탭에는 기사 분석 대신 미충족 사유, 탐색한 출처 범위, 대체하지 않은 이유, 다음 확인 필요 사항을 표시한다.
* 미충족 탭에서는 임의 기사 링크를 생성하지 않는다.
* 리포트 생성 후 반드시 렌더링 필수 데이터 검사를 수행한다.
* 각 헤더 탭의 `areas`, `evidence`, `cases`는 배열이어야 하며, 각 영향 영역에는 `scope`, `why`, `actions`, `indicators`, `impactPath`, `directImpact`, `finalRisk`, `insight`가 있어야 한다.
* `상세 분석`, `근거 자료`, `권고 액션`, `유사·참고 사례` 중 하나라도 빈 화면이 되거나 렌더링 오류가 발생하면 리포트를 완료한 것으로 보지 않는다. 원인을 수정한 뒤 다시 검사한다.
* `outputs/template.html`의 렌더링 JS는 데이터 누락이 있더라도 섹션 전체가 멈추지 않도록 기본값을 제공해야 하며, 데일리 리포트 생성 시 이 방어 로직을 삭제하거나 우회하지 않는다.
* 기존 리포트의 기사, 날짜, 분석 내용, 근거 자료, 권고 액션을 재사용하지 않는다.
* 기존 리포트와 중복되는 기준 기사 또는 같은 사건을 다시 사용하지 않으며, 중복 점검 기준을 통과하지 못하면 파일을 생성하지 않는다.
* 날짜는 실행일 기준 `YYYY-MM-DD`로 표시한다.
* 출력 파일명은 실행일 기준 `outputs/데일리_인사시사점리포트_YYYYMMDD.html` 형식을 사용한다.
* 매일 리포트 생성 후 `outputs/latest.html`을 같은 내용으로 갱신한다.
* 매일 리포트 생성 후 `outputs/reports.json`에 해당 날짜의 이력 항목을 추가하거나 갱신한다.
* `outputs/reports.json`의 각 항목은 `date`, `title`, `file`, `latest`, `issues` 값을 포함한다.
* 6개 헤더 구조에서는 `issues`에 업계, 산업, 관계사, 정책, 경제, 글로벌 헤더별 제목 또는 미충족 상태가 모두 포함되어야 한다.
* 새 리포트를 추가할 때 기존 날짜의 이력 항목은 삭제하지 않는다.
* 같은 날짜의 리포트를 재생성하면 기존 항목을 중복 추가하지 말고 같은 날짜 항목을 갱신한다.
* `archive.html`은 날짜별 이력 달력 화면이므로, 리포트 목록 구조를 바꿀 때 함께 동작 여부를 확인한다.
* 프로젝트 루트와 기존 파일 구조를 유지하며, 새 프로젝트를 만들지 않는다.

데일리 리포트 데이터의 `issues` 배열은 아래 순서를 유지한다.

3. 관계사
4. 정책
5. 경제

## 렌더링 및 검증 기준

리포트 생성 후 아래 항목을 반드시 확인한다.

* 6개 탭이 모두 표시되는가
* 탭 순서가 `업계`, `산업`, `관계사`, `정책`, `경제`, `글로벌` 순서인가
* 최초 진입 시 `업계` 탭이 기본 선택되어 있는가
* 기사 제목 앞에 `[업계]`, `[산업]`, `[관계사]`, `[정책]`, `[경제]`, `[글로벌]` 형식의 헤더가 표시되는가
* 각 탭 클릭 시 해당 헤더의 기사 또는 미충족 상태가 정상 표시되는가
* 선정된 기사들의 보도일이 실행일과 같은가
* 미충족 탭이 있는 경우 과거 기사나 임의 링크로 대체되지 않았는가
* 각 완료 탭의 People 기능 9개가 모두 표시되는가
* 각 완료 탭의 상위 리스크 3개가 표시되는가
* 각 완료 탭의 근거 자료, 권고 액션, 유사·참고 사례가 비어 있지 않은가
* 각 완료 탭의 링크 버튼이 실제 출처 URL을 사용하는가
* `outputs/latest.html`이 날짜별 HTML과 같은 내용으로 갱신되었는가
* `outputs/reports.json`에 해당 날짜 항목이 추가 또는 갱신되었는가
* `archive.html`에서 해당 날짜의 6개 헤더별 이슈가 확인되는가

## 미충족 처리 기준

특정 헤더에서 실행일 당일 적합한 기사를 확보하지 못하면 해당 헤더를 `당일 기사 기준 미충족`으로 표시한다.

미충족 처리 시 반드시 포함할 내용은 아래와 같다.

* 미충족 헤더명
* 미충족 사유
* 탐색한 출처 범위
* 과거 기사로 대체하지 않은 이유
* 다음 확인 필요 사항

미충족 헤더가 있어도 6개 탭 구조는 유지한다.

단, 미충족 헤더가 과도하게 많아 리포트의 실질적 가치가 낮다고 판단되면 파일을 생성하지 않고 `당일 기사 기준 미충족`으로 보고할 수 있다.

특히 아래 조건 중 하나에 해당하면 리포트 발행을 보류한다.

* 국내 완료 기사 수가 3건 미만인 경우
* 업계, 관계사, 정책, 경제 중 3개 이상이 미충족인 경우
* 글로벌 기사만 있고 국내 People 영향이 있는 기사가 거의 없는 경우
* 기준 기사 대부분이 단순 시장 전망, 실적 전망, 사설/칼럼에 가까운 경우
